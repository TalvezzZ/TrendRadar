## 系统概述

Claude Code 的记忆系统是一个多层次、持久化的上下文管理系统，旨在让 Claude 在未来的对话中记住用户的偏好、项目背景和重要信息。

### 核心特性
- 📝 **类型化记忆**：四种明确定义的记忆类型
- 💾 **持久化存储**：文件系统为基础的存储
- 🔍 **智能检索**：基于相关性的自动加载
- 🤝 **团队共享**：支持团队级别的记忆同步
- 🤖 **自动提取**：后台Agent自动提取关键信息
- 📊 **会话总结**：自动维护会话笔记

### 记忆系统架构

```
Claude Code 记忆系统
├── Auto Memory (自动记忆)
│   ├── MEMORY.md (索引文件)
│   ├── user_*.md (用户记忆)
│   ├── feedback_*.md (反馈记忆)
│   ├── project_*.md (项目记忆)
│   ├── reference_*.md (参考记忆)
│   └── logs/ (Assistant模式日志)
│       └── YYYY/MM/YYYY-MM-DD.md
├── Session Memory (会话记忆)
│   └── session_notes.md
├── Agent Memory (代理记忆)
│   └── <agent-type>/
│       └── MEMORY.md
└── Team Memory (团队记忆)
    └── team/
        └── MEMORY.md
```

## 记忆类型

### 1. User (用户记忆)
**用途**：记录用户的角色、目标、责任和知识

**Scope**: 始终私有

**何时保存**：
- 了解用户的角色、偏好、责任或知识时

**如何使用**：
- 根据用户的个人资料或视角调整工作
- 为不同经验层次的用户定制解释

**示例**：
```markdown
---
name: user_role
description: User is a data scientist focused on observability
type: user
---

User is a data scientist currently investigating logging infrastructure.

**Why**: User explicitly stated their role and current focus area.

**How to apply**: Frame technical explanations in terms of data analysis and monitoring concepts. Prioritize observability-related suggestions.
```

### 2. Feedback (反馈记忆)
**用途**：记录用户关于工作方式的指导（避免什么和保持什么）

**Scope**: 
- 默认私有
- 仅在明确是项目范围的约定时保存为团队记忆（如测试政策、构建约束）

**何时保存**：
- 用户纠正方法时（"不要那样"，"停止做X"）
- 用户确认非显而易见的方法有效时（"是的，就这样"，"完美，继续这样做"）

**结构要求**：
```markdown
规则本身

**Why**: 原因（通常是过去的事件或强烈偏好）

**How to apply**: 何时/何处应用此指导
```

**示例**：
```markdown
---
name: feedback_testing
description: Integration tests must use real database
type: feedback
---

Integration tests must hit a real database, not mocks.

**Why**: Last quarter mocked tests passed but prod migration failed due to mock/prod divergence.

**How to apply**: When writing integration tests for database operations, always use a test database instance rather than mocking the database layer.
```

### 3. Project (项目记忆)
**用途**：记录无法从代码或Git历史推导的正在进行的工作、目标、计划、Bug或事件

**Scope**: 私有或团队，强烈倾向于团队

**何时保存**：
- 了解谁在做什么、为什么、何时时
- **重要**：总是将相对日期转换为绝对日期（如 "Thursday" → "2026-03-05"）

**结构要求**：
```markdown
事实或决定

**Why**: 动机（通常是约束、截止日期或利益相关者要求）

**How to apply**: 这应该如何影响建议
```

**示例**：
```markdown
---
name: project_merge_freeze
description: Merge freeze for mobile release branch
type: project
---

Merge freeze begins 2026-03-05 for mobile team release cut.

**Why**: Mobile team is cutting a release branch and needs stability.

**How to apply**: Flag any non-critical PR work scheduled after 2026-03-05. Only critical bugfixes should be merged during freeze.
```

### 4. Reference (参考记忆)
**用途**：存储外部系统中信息位置的指针

**Scope**: 通常为团队

**何时保存**：
- 了解外部系统中的资源及其用途时
- 例如：Linear项目、Slack频道、Grafana仪表板

**示例**：
```markdown
---
name: reference_bug_tracking
description: Pipeline bugs tracked in Linear INGEST project
type: reference
---

Pipeline bugs are tracked in Linear project "INGEST".

**Why**: This is the canonical source for all pipeline-related issues.

**How to apply**: When investigating or reporting pipeline bugs, check INGEST project first.
```

## 不应保存的内容

以下内容**不应**保存到记忆中：

1. **代码模式、约定、架构、文件路径或项目结构**
   - 这些可以通过读取当前项目状态推导

2. **Git历史、最近更改、谁改了什么**
   - `git log` / `git blame` 是权威来源

3. **调试解决方案或修复配方**
   - 修复在代码中；提交消息有上下文

4. **CLAUDE.md文件中已记录的任何内容**

5. **临时任务细节**
   - 正在进行的工作、临时状态、当前对话上下文

**重要**：即使用户明确要求保存，这些排除也适用。如果用户要求保存PR列表或活动摘要，询问其中*令人惊讶*或*非显而易见*的部分——这才是值得保留的。

## 自动记忆(Auto Memory)

### 概述
自动记忆是主要的持久化记忆系统，跨会话保存用户偏好、项目上下文和重要决策。

### 存储结构

#### 标准模式
```
~/.claude/projects/<sanitized-git-root>/memory/
├── MEMORY.md                 # 索引文件（总是加载）
├── user_role.md              # 用户记忆
├── feedback_testing.md       # 反馈记忆
├── project_deadline.md       # 项目记忆
└── reference_dashboards.md   # 参考记忆
```

#### Assistant模式（KAIROS）
```
~/.claude/projects/<sanitized-git-root>/memory/
├── MEMORY.md                 # 提炼的索引（每晚更新）
└── logs/
    └── 2026/
        └── 04/
            └── 2026-04-07.md  # 每日追加日志
```

### 文件格式

#### Frontmatter格式
```markdown
---
name: {{记忆名称}}
description: {{单行描述 - 用于决定相关性，要具体}}
type: {{user, feedback, project, reference}}
---

{{记忆内容 - feedback/project类型，结构为：规则/事实，然后是 **Why:** 和 **How to apply:** 行}}
```

#### MEMORY.md索引格式
```markdown
- [Title](file.md) — 单行摘要（约150字符以内）
```

### 限制

```typescript
const MAX_ENTRYPOINT_LINES = 200      // 最大200行
const MAX_ENTRYPOINT_BYTES = 25_000   // 最大25KB
```

超出限制会截断并显示警告。

### 启用/禁用

优先级链（第一个定义的生效）：

1. `CLAUDE_CODE_DISABLE_AUTO_MEMORY` 环境变量
   - `1`/`true` → 关闭
   - `0`/`false` → 开启

2. `CLAUDE_CODE_SIMPLE` (--bare) → 关闭

3. CCR无持久存储 → 关闭（无`CLAUDE_CODE_REMOTE_MEMORY_DIR`）

4. `autoMemoryEnabled` 在 settings.json

5. 默认：**启用**

### 两步保存流程

**Step 1**：写入单独的文件
```bash
# 使用Write工具创建
memory/feedback_testing.md
```

**Step 2**：添加指针到MEMORY.md
```markdown
- [Testing Policy](feedback_testing.md) — Always use real DB in integration tests
```

### Assistant模式日志

在Assistant模式（长期会话）中：
- 新记忆追加到每日日志：`logs/YYYY/MM/YYYY-MM-DD.md`
- 日志是追加式的，不重写或重组
- 每晚的 `/dream` 技能将日志提炼为主题文件和MEMORY.md

日志格式：
```markdown
- 09:30 User prefers bun over npm
- 14:45 Project deadline moved to 2026-04-15 due to compliance review
- 16:20 Dashboard at grafana.internal/d/api-latency used by oncall
```

## 会话记忆(Session Memory)

### 概述
会话记忆自动维护当前对话的Markdown笔记，使用后台Agent定期提取关键信息。

### 存储位置
```
~/.claude/projects/<project-slug>/session_notes.md
```

### 工作机制

#### 1. 初始化阈值
```typescript
DEFAULT_CONFIG = {
  minimumMessageTokensToInit: 100_000,      // 初始化需要100K tokens
  minimumTokensBetweenUpdate: 50_000,       // 更新间隔50K tokens
  toolCallsBetweenUpdates: 50               // 或50次工具调用
}
```

#### 2. 提取触发条件
```typescript
shouldExtract = 
  (hasMetTokenThreshold && hasMetToolCallThreshold) ||
  (hasMetTokenThreshold && !hasToolCallsInLastTurn)
```

满足以下任一条件触发提取：
- Token阈值 **且** 工具调用阈值都满足
- Token阈值满足 **且** 最后一轮无工具调用（自然对话断点）

**重要**：Token阈值是**必需**的，即使工具调用阈值满足也不会提取。

#### 3. 提取流程
```typescript
extractSessionMemory(context) {
  if (querySource !== 'repl_main_thread') return  // 仅主线程
  if (!isSessionMemoryGateEnabled()) return        // 功能门控
  if (!shouldExtractMemory(messages)) return       // 阈值检查
  
  markExtractionStarted()
  
  // 1. 设置文件系统和读取当前状态
  const { memoryPath, currentMemory } = await setupSessionMemoryFile()
  
  // 2. 构建提取提示词
  const userPrompt = await buildSessionMemoryUpdatePrompt(currentMemory, memoryPath)
  
  // 3. 运行forked agent进行提取
  await runForkedAgent({
    promptMessages: [userPrompt],
    canUseTool: createMemoryFileCanUseTool(memoryPath),  // 仅允许Edit该文件
    querySource: 'session_memory',
    forkLabel: 'session_memory'
  })
  
  markExtractionCompleted()
}
```

#### 4. 权限限制
```typescript
createMemoryFileCanUseTool(memoryPath) {
  return async (tool, input) => {
    // 仅允许对指定文件使用FileEditTool
    if (tool.name === 'FileEditTool' && input.file_path === memoryPath) {
      return { behavior: 'allow' }
    }
    return { behavior: 'deny' }
  }
}
```

### 模板结构

初始session_notes.md包含：
- 会话目标
- 关键决策
- 重要发现
- 待办事项
- 上下文和约束

### 手动提取

通过 `/summary` 命令触发：
```typescript
manuallyExtractSessionMemory(messages, toolUseContext)
  // 绕过阈值检查，立即提取
```

### 与Auto-compact集成
会话记忆用于自动压缩：
```typescript
if (!isAutoCompactEnabled()) {
  return  // 不启用会话记忆
}
```

## Agent记忆

### 概述
Agent记忆为每个agent类型提供持久化的学习能力，允许专门的agent在多次调用间保留知识。

### 作用域（Scope）

#### 1. User Scope (用户范围)
```
~/.claude/agent-memory/<agent-type>/MEMORY.md
```
- 跨所有项目共享
- 适合通用学习

#### 2. Project Scope (项目范围)
```
<cwd>/.claude/agent-memory/<agent-type>/MEMORY.md
```
- 项目特定
- 通过版本控制与团队共享

#### 3. Local Scope (本地范围)
```
<cwd>/.claude/agent-memory-local/<agent-type>/MEMORY.md
```
- 项目特定但不纳入版本控制
- 机器/环境特定的配置

```typescript
function sanitizeAgentTypeForPath(agentType: string): string {
```

## Team记忆

### 概述
Team记忆扩展了自动记忆系统，添加了团队范围的共享记忆目录。

### 存储结构
```
~/.claude/projects/<project>/memory/
├── MEMORY.md                    # 私有记忆索引
├── user_*.md                    # 私有记忆
├── feedback_private.md          # 私有反馈
└── team/                        # 团队目录
    ├── MEMORY.md                # 团队记忆索引
    ├── feedback_project.md      # 团队反馈
    ├── project_*.md             # 团队项目记忆
    └── reference_*.md           # 团队参考
```

#### User记忆
- **始终私有**

#### Feedback记忆
- **默认私有**
- 仅在明确是项目范围约定时保存为团队
  - 例如：测试政策、构建约束
  - 非个人风格偏好

#### Project记忆
- **强烈倾向团队**
- 除非明确是个人进度追踪

#### Reference记忆
- **通常为团队**

## 记忆检索与相关性

### 智能检索流程

#### 1. 扫描记忆文件
```typescript
scanMemoryFiles(memoryDir, signal) {
  // 1. 递归读取.md文件（排除MEMORY.md）
  // 2. 读取每个文件的frontmatter（最多30行）
  // 3. 提取name, description, type, mtimeMs
  // 4. 按修改时间排序（最新优先）
  // 5. 限制200个文件
  
  return MemoryHeader[]
}
```

#### 2. 相关性选择
```typescript
findRelevantMemories(query, memoryDir, signal, recentTools, alreadySurfaced) {
  // 1. 扫描记忆文件
  const memories = await scanMemoryFiles(memoryDir)
  
  // 2. 过滤已展示的
  const filtered = memories.filter(m => !alreadySurfaced.has(m.filePath))
  
  // 3. 使用Sonnet选择相关的（最多5个）
  const selected = await selectRelevantMemories(query, filtered, recentTools)
  
  return selected.map(m => ({ path: m.filePath, mtimeMs: m.mtimeMs }))
}
```

#### 3. Sonnet选择器

**系统提示**：
```
You are selecting memories that will be useful to Claude Code as it processes a user's query.

Return a list of filenames (up to 5). Only include memories that you are certain will be helpful.
- If unsure, do not include it. Be selective and discerning.
- If no memories would clearly be useful, return an empty list.
- If recently-used tools list is provided, do not select usage/API docs for those tools 
  (Claude is already using them). DO select warnings/gotchas/known issues.
```

**输入格式**：
```
Query: <user query>

Available memories:
- [type] filename (timestamp): description
- [feedback] testing_policy.md (2026-04-03T10:00:00Z): Integration tests must use real DB
- [project] deadline.md (2026-04-01T14:00:00Z): Launch moved to 2026-04-15
...

Recently used tools: BashTool, FileReadTool
```

**输出格式**（JSON Schema）：
```json
{
  "selected_memories": ["testing_policy.md", "deadline.md"]
}
```

#### 4. 工具过滤
```typescript
// 避免选择正在使用的工具的文档
const toolsSection = recentTools.length > 0
  ? `\n\nRecently used tools: ${recentTools.join(', ')}`
  : ''
```

如果Claude正在使用`mcp__github__create_pr`，不要选择该工具的参考文档（已在对话中有使用示例）。但**仍然**选择关于该工具的警告、陷阱或已知问题。

### 记忆清单格式
```typescript
formatMemoryManifest(memories: MemoryHeader[]): string {
  return memories.map(m => {
    const tag = m.type ? `[${m.type}] ` : ''
    const ts = new Date(m.mtimeMs).toISOString()
    return m.description
      ? `- ${tag}${m.filename} (${ts}): ${m.description}`
      : `- ${tag}${m.filename} (${ts})`
  }).join('\n')
}
```

### /memory 命令

#### 功能

```typescript
3. 显示选择对话框
4. 用户选择文件
5. 在编辑器中打开（使用$EDITOR或$VISUAL）
6. 保存后自动重新加载
```

```tsx

```

#### 编辑器选择

## 记忆访问指导

### 何时访问记忆

1. **记忆似乎相关时**，或用户引用先前对话的工作
2. **必须访问**：用户明确要求检查、回忆或记住
3. **忽略指令**：用户说*忽略*或*不使用*记忆时
   - 像MEMORY.md为空一样进行
   - 不应用记忆的事实，不引用，不比较，不提及记忆内容

### 验证记忆
```
记忆记录会随时间变旧。在回答用户或基于记忆构建假设前：
1. 验证记忆仍然正确和最新
2. 读取文件/资源的当前状态
3. 如果记忆与当前信息冲突，信任观察到的现在
4. 更新或删除过时的记忆，而不是基于它行动
```

### 推荐前验证
```
记忆中命名的函数、文件或标志是*当时写入时*存在的声明。
它可能已被重命名、删除或从未合并。

推荐前：
- 记忆命名文件路径：检查文件存在
- 记忆命名函数或标志：grep查找
- 用户即将基于推荐行动：先验证

"记忆说X存在" ≠ "X现在存在"
```

### 快照记忆
```
总结仓库状态的记忆（活动日志、架构快照）是冻结在时间点的。

如果用户询问*最近*或*当前*状态，优先使用`git log`或读取代码，
而非回忆快照。
```

## 记忆与其他持久化机制

### Plan vs Memory
```typescript
何时使用Plan而非Memory：
- 即将开始非平凡的实现任务
- 需要与用户就方法达成一致
- 已有计划并更改了方法

Plan是对话内的，Memory是跨对话的
```

### Task vs Memory
```typescript
何时使用Task而非Memory：
- 需要将当前对话的工作分解为离散步骤
- 跟踪当前对话的进度

Task用于当前对话的工作，Memory用于未来对话的信息
```

## 最佳实践

### 1. 记忆内容质量

#### ✅ 好的记忆
```markdown
---
name: feedback_pr_strategy
description: For refactors in auth module, prefer bundled PRs
type: feedback
---

For refactors in the auth module, prefer single bundled PR over many small ones.

**Why**: User confirmed after I chose this approach - splitting would create churn without adding review value in this area.

**How to apply**: When refactoring authentication code, bundle related changes into coherent PRs rather than splitting by file or type of change.
```

#### ❌ 差的记忆
```markdown
---
name: stuff
description: Some changes
type: feedback
---

User said to do it differently. Make better PRs.
```

**问题**：
- 名称不描述性
- 描述太模糊
- 缺少Why和How to apply
- 无法判断何时应用

### 2. 描述质量

描述用于相关性选择，必须**具体**：

✅ `Integration tests must hit real database due to mock/prod divergence incident`

❌ `Testing policy`

### 3. 记忆组织

#### 按主题组织
```
memory/
├── user_role.md
├── user_preferences.md
├── feedback_testing_db.md
├── feedback_pr_strategy.md
├── project_auth_rewrite.md
├── project_mobile_freeze.md
└── reference_linear_projects.md
```

#### 不要按时间组织
```
❌ memory/
    ├── 2026-04-01.md
    ├── 2026-04-02.md
    └── 2026-04-03.md
```

### 4. 更新vs创建

在创建新记忆前：
1. 检查是否有相关的现有记忆
2. 如果有，更新现有记忆
3. 避免重复记忆

### 5. 删除过时记忆

定期审查和删除：
- 不再相关的项目记忆（项目完成）
- 已过期的截止日期
- 不再使用的工具参考
- 过时的架构决策

### 6. 索引简洁性

MEMORY.md限制：
- 200行
- 25KB
- 每条~150字符

**保持简洁**：
```markdown
✅ - [DB Testing](feedback_db.md) — Always use real DB, not mocks

❌ - [Database Testing Policy and Guidelines](feedback_database_testing_comprehensive.md) — This is a comprehensive policy document that outlines our approach to database testing in integration test suites, which was established after an incident last quarter where...
```

### 7. Type字段的使用

**总是**指定正确的type：
```markdown
---
type: user      # 用户信息
type: feedback  # 工作方式指导
type: project   # 正在进行的工作上下文
type: reference # 外部系统指针
---
```

这有助于：
- 相关性选择
- Scope决策（private vs team）
- 记忆分类和组织

### 8. Scope决策流程图

```
记忆是关于用户个人的吗？
├─ 是 → private (user type)
└─ 否 → 继续

记忆是个人沟通偏好吗？
├─ 是 → private (feedback type)
└─ 否 → 继续

记忆是项目范围的约定/政策吗？
├─ 是 → team (feedback/project/reference type)
└─ 否 → 继续

记忆是个人进度追踪吗？
├─ 是 → private (project type)
└─ 否 → team (project type - 默认)
```

### 9. 日期处理

**总是**转换相对日期为绝对日期：

```markdown
✅ Merge freeze begins 2026-04-05 for mobile release

❌ Merge freeze begins this Thursday for mobile release
```

相对日期在未来会话中无法解释。

### 10. 反馈记忆的结构

**模板**：
```markdown
[规则/指导]

**Why**: [原因 - 通常是事件或偏好]

**How to apply**: [何时/何处应用]
```

**示例**：
```markdown
Don't mock the database in integration tests.

**Why**: Last quarter mocked tests passed but prod migration failed due to mock/prod divergence.

**How to apply**: When writing integration tests that touch the database, spin up a test database instance rather than mocking the database layer. This applies to all tests in tests/integration/.
```

### 常见问题

#### 1. 记忆未加载
```bash
# 检查路径
echo "Auto mem path: $(cat ~/.claude/config.json | jq -r .autoMemoryDirectory)"

# 检查启用状态
grep "autoMemoryEnabled" ~/.claude/settings.json

# 检查环境变量
env | grep CLAUDE_CODE_DISABLE_AUTO_MEMORY
```

#### 2. MEMORY.md被截断
```markdown
> WARNING: MEMORY.md is 250 lines (limit: 200). 
> Only part of it was loaded. Keep index entries to one line 
> under ~200 chars; move detail into topic files.
```

**解决**：
- 缩短索引条目
- 将详细内容移到主题文件
- 删除过时条目

#### 3. 记忆选择不相关
**原因**：描述太模糊

**解决**：改进描述的具体性
```markdown
❌ description: Testing stuff
✅ description: Integration tests must use real DB due to mock divergence
```

## 总结

Claude Code的记忆系统是一个复杂而强大的多层次持久化系统：

1. **Auto Memory**: 主要的跨会话记忆，四种类型化记忆
2. **Session Memory**: 自动维护的会话笔记
3. **Agent Memory**: Agent特定的持久学习
4. **Team Memory**: 团队共享的记忆同步

通过智能检索、类型化结构和最佳实践，记忆系统让Claude能够：
- 记住用户偏好和工作方式
- 保留项目上下文和决策
- 跨会话持续学习
- 与团队共享知识

正确使用记忆系统可以显著提升Claude的协作效率和上下文连续性。
