```

#### Claude保存反馈记忆

**feedback_testing_database.md**
```markdown
---
name: feedback_no_db_mocks
description: Never mock database in integration tests
type: feedback
---

Integration tests MUST use a real database, never mocks.

**Why**: Last quarter, mocked tests passed but production migration failed due to mock/prod divergence. This caused a critical incident.

**How to apply**: 
- Integration tests in `tests/integration/`: spin up test database
- Use docker-compose for local test DB
- In CI: use test database service
- Only unit tests for business logic can mock the DB layer
```

### 案例4：Team记忆共享最佳实践

**场景**：团队形成了一个新的代码审查约定。

#### 团队讨论后
```
User: 我们团队决定，超过500行的PR必须拆分，除非是重构
```

#### 保存为Team记忆

**team/feedback_pr_size_policy.md**
```markdown
---
name: feedback_pr_size_limit
description: PRs over 500 lines must be split unless refactoring
type: feedback
---

PRs exceeding 500 lines must be split into smaller PRs.

Exception: Refactoring that needs to be atomic can stay bundled.

**Why**: Team consensus on 2026-04-05. Large PRs are hard to review and increase merge conflict risk.

**How to apply**:
- Before creating PR, check line count: `git diff --stat main...HEAD`
- If >500 lines and not refactoring, split by:
  - Logical feature boundaries
  - File groups
  - Dependent changes in sequence
- Refactoring that spans files but must be atomic: keep together but add explanation in PR description
```

#### 其他团队成员获益

团队成员B与Claude的对话：
```
User B: 帮我创建这个新功能的PR
Claude: [检查team记忆]

我注意到这个改动大约800行。根据团队的PR大小政策（team/feedback_pr_size_policy.md），我们应该拆分它。

建议拆分方案：
1. PR #1: 数据模型更改 (models/ + migrations/)
2. PR #2: API端点实现 (routes/ + controllers/)
3. PR #3: 前端组件 (components/)

这样每个PR都在500行以内，更容易审查。
```

## 高级用法

### 1. 条件性记忆（基于上下文）

**技巧**：在记忆中包含条件应用逻辑。

**示例**：
```markdown
---
name: feedback_error_handling_context
description: Error handling strategy varies by context
type: feedback
---

Error handling strategy depends on the context:

**API Routes** (`src/routes/`):
- Always catch and return HTTP error codes
- Log to Sentry
- Return user-friendly messages

**Background Jobs** (`src/jobs/`):
- Let errors bubble up
- Job queue handles retries
- Only catch for cleanup (close connections, etc.)

**Utilities** (`src/utils/`):
- Throw specific error types
- Let caller decide handling
- Document thrown errors in JSDoc

**Why**: Different contexts have different error handling needs and retry mechanisms.

**How to apply**: Check file path to determine context, then apply appropriate pattern.
```

### 2. 链接记忆（交叉引用）

**技巧**：记忆之间可以相互引用。

**primary记忆 - project_auth_rewrite.md**：
```markdown
---
name: project_auth_migration
description: OAuth2 migration in progress, deadline 2026-05-01
type: project
---

Migrating from custom JWT to OAuth2 (Auth0).

Deadline: 2026-05-01 (compliance requirement)

See also: 
- feedback_auth_testing.md for testing approach
- reference_auth0_docs.md for Auth0 setup

**Why**: Legal requirement for SOC2 compliance.

**How to apply**: All new auth code should use OAuth2. Legacy JWT code in deprecation mode.
```

**linked记忆 - feedback_auth_testing.md**：
```markdown
---
name: feedback_auth_testing_strategy
description: Auth testing must include both new OAuth2 and legacy JWT
type: feedback
---

During auth migration (see project_auth_migration.md), tests must cover both:
1. New OAuth2 flows
2. Legacy JWT (until migration complete)

**Why**: Dual-mode operation during migration period.

**How to apply**: Until 2026-05-01, all auth tests should have JWT and OAuth2 variants.
```

### 3. 版本化记忆（追踪变化）

**技巧**：在记忆内容中追踪历史变化。

```markdown
---
name: project_deployment_strategy
description: Deployment strategy (updated 2026-04-05)
type: project
---

## Current Strategy (as of 2026-04-05)

Blue-green deployment with manual cutover.

**Why**: Allows instant rollback if issues detected.

**How to apply**: Deploy to green, smoke test, manual switch, monitor.

---

## Previous Strategy (until 2026-03-15)

Rolling deployment with automatic cutover.

**Deprecated because**: Had a production incident where automatic cutover happened during peak traffic, causing cascade failure.

---

## Future Strategy (planned 2026-06-01)

Canary deployment with gradual traffic shifting.

**Why**: Better than blue-green for catching edge cases.
```

### 4. 分层记忆（通用→具体）

**技巧**：从通用原则到具体实现分层组织。

**通用原则 - feedback_code_quality_principles.md**：
```markdown
---
name: feedback_code_quality
description: General code quality principles for this project
type: feedback
---

1. Prefer composition over inheritance
2. No God objects (max 300 lines per class)
3. Fail fast with clear error messages
4. Test coverage >80% for core business logic

**Why**: Team consensus, established in architecture review 2025-12.

**How to apply**: All new code should follow these principles.
```

**具体实现 - feedback_react_patterns.md**：
```markdown
---
name: feedback_react_component_patterns
description: React-specific patterns (extends feedback_code_quality)
type: feedback
---

Extends: feedback_code_quality.md

React-specific:
1. Composition: Prefer component composition over render props
2. Size limit: Max 200 lines per component (stricter than general 300)
3. Error boundaries: Wrap route components
4. Testing: Use React Testing Library, test user behavior not implementation

**Why**: React-specific best practices aligned with general principles.

**How to apply**: When writing React components.
```

### 5. 临时记忆（带过期日期）

**技巧**：为临时决策添加过期提醒。

```markdown
---
name: project_temp_workaround_api_rate_limit

## 疑难场景处理

### 场景1：记忆冲突

**问题**：私有记忆和团队记忆冲突。

**团队记忆**：
```markdown
team/feedback_testing.md
---
All PRs must have >80% test coverage
---
```

**私有记忆**（用户的偏好）：
```markdown
feedback_testing_my_approach.md
---
For experimental features, I prefer to iterate fast first, add tests later.

**Override**: This overrides team policy for my personal experimental branches.

**How to apply**: On branches named `experiment/*`, lower test coverage is OK until feature is validated.
---
```

**Claude的处理**：
```
User: 创建实验性功能分支
Claude: 我注意到这是`experiment/`分支。

团队政策要求80%覆盖率（team/feedback_testing.md），
但你的私有偏好（feedback_testing_my_approach.md）是实验分支先快速迭代。

我会遵循你的偏好：先实现功能，验证后再添加测试。
合并到main前，我们需要补充测试到80%覆盖率。
```

### 场景2：记忆过载（太多记忆）

**问题**：记忆文件太多，选择效率低。

**症状**：
- 200+个记忆文件
- 相关性选择缓慢
- 经常选错

**解决方案**：

1. **合并相似记忆**
   ```
   Before: 
   - feedback_react_hooks_1.md
   - feedback_react_hooks_2.md
   - feedback_react_hooks_3.md
   
   After:
   - feedback_react_best_practices.md (综合)
   ```

2. **归档旧记忆**
   ```bash
   mkdir memory/archive/
   mv memory/project_2025_*.md memory/archive/
   ```

3. **改进描述质量**
   ```markdown
   ❌ description: React stuff
   ✅ description: React hooks must handle cleanup in useEffect return
   ```

```markdown

```typescript
// 选择器可以优先选择特定类型
query: "what's the user's background?"
→ 优先选择type: user的记忆

query: "what's our testing policy?"
→ 优先选择type: feedback的记忆

query: "what's the deadline?"
→ 优先选择type: project的记忆
```

## 实用工具脚本

### 脚本1：记忆统计

```bash
#!/bin/bash
# memory-stats.sh

MEMORY_DIR="$HOME/.claude/projects/$(basename $PWD)/memory"

echo "=== Memory Statistics ==="
echo "Total files: $(find "$MEMORY_DIR" -name '*.md' | wc -l)"
echo ""

echo "By type:"
echo "User:      $(grep -r "^type: user$" "$MEMORY_DIR" | wc -l)"
echo "Feedback:  $(grep -r "^type: feedback$" "$MEMORY_DIR" | wc -l)"
echo "Project:   $(grep -r "^type: project$" "$MEMORY_DIR" | wc -l)"
echo "Reference: $(grep -r "^type: reference$" "$MEMORY_DIR" | wc -l)"
echo ""

echo "Index stats:"
INDEX="$MEMORY_DIR/MEMORY.md"
if [ -f "$INDEX" ]; then
  echo "Lines: $(wc -l < "$INDEX")"
  echo "Bytes: $(wc -c < "$INDEX")"
  echo "Limit: 200 lines / 25000 bytes"
fi
```

### 脚本2：查找过时记忆

```bash
#!/bin/bash
# find-stale-memories.sh

MEMORY_DIR="$HOME/.claude/projects/$(basename $PWD)/memory"
DAYS=90  # 90天未修改

echo "=== Memories not modified in $DAYS days ==="
find "$MEMORY_DIR" -name '*.md' -type f -mtime +$DAYS -exec ls -lh {} \;
```

### 脚本3：记忆备份

```bash
#!/bin/bash
# backup-memories.sh

MEMORY_DIR="$HOME/.claude/projects/$(basename $PWD)/memory"
BACKUP_DIR="$HOME/.claude/backups/$(date +%Y%m%d)"

mkdir -p "$BACKUP_DIR"
cp -r "$MEMORY_DIR" "$BACKUP_DIR/"

echo "Backed up to: $BACKUP_DIR"
```

### 脚本4：检查记忆引用完整性

```bash
#!/bin/bash
# check-memory-refs.sh

MEMORY_DIR="$HOME/.claude/projects/$(basename $PWD)/memory"

echo "=== Checking memory cross-references ==="

# 找到所有.md引用
grep -rh "See also:" "$MEMORY_DIR" | sed 's/.*See also://' | tr ',' '\n' | while read ref; do
  ref=$(echo "$ref" | xargs)  # trim
  if [ ! -f "$MEMORY_DIR/$ref" ]; then
    echo "❌ Broken reference: $ref"
  fi
done

echo "Done"
```

## 总结

记忆系统的强大之处在于：
1. **持久性**：跨会话保留上下文
2. **智能性**：自动选择相关记忆
3. **协作性**：团队共享知识
4. **灵活性**：适应不同场景

掌握记忆系统的关键：
- ✅ **高质量的描述**：specific > generic
- ✅ **结构化的内容**：Why + How to apply
- ✅ **及时的维护**：定期审查和清理
- ✅ **合理的scope**：私有vs团队的正确划分

记忆系统不是被动的存储，而是主动的知识管理伙伴。善用它，Claude会成为真正了解你和项目的智能助手。
